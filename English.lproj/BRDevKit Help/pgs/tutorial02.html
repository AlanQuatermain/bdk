Step 2: Setting Up the Source

Having created the sample project, there are a few preliminary steps we should take care of now, to avoid conflicts later.

Since this is a plugin loaded into another application along with a potentially large list of other plugins, we need to take into account the fact that there are no namespaces in Objective-C. The standard workaround for this is to use unique prefixes for your class names; the base OS X classes all use 'NS' (from NeXTStep), BackRow uses BR and in some cases RUI, and I personally favour AQ. Your initials, or the initials of the project you're working on are usually a good choice. In this tutorial, we will be using 'BDK'.

Open TutorialAppliance.h, and change the class name to BDKTutorialAppliance

Open TutorialAppliance.m and make the same change. Note that you'll also need to change the value in the -moduleKey function, and make a similar change for the controller class name in -applianceControllerForScene:.

Open TutorialApplianceController.h and TutorialApplianceController.m, making the same changes to the class names.

Your two classes should now be correctly named BDKTutorialAppliance and BDKTutorialApplianceController. If you attempt to compile the project now, if should compile without error.

Our next order of business is the bundle identifier, which should be changed to something unique.

Open Info.plist; preferably by control-clicking (or right-clicking) on it and choosing 'Open With Finder' to open it using the Property List Editor application.

Edit the value for the CFBundleIdentifier key so that it reads com.yourcompany.bdktutorial.

Edit the value for NSPrincipalClass, placing BDK in front of the name, to match our new class name.

Open Appliance.m, and edit the value returned from +moduleKey to also read com.yourcompany.bdktutorial.
